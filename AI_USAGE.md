# How I used AI on this take-home

This project was built with Claude Code (Opus 4.7). The transcript is below in the form
that matters: **what the AI proposed, where I disagreed or verified, and what I owned.**

If you only read one section, read §3 — that's where the value of an experienced engineer
over a literal AI execution shows up.

## §1 — Architecture

**AI's first instinct:** When I gave Claude the sujet, its first proposal was a
monolithic `buildPostCallDecision` with a big `switch/if-else` cascade. Simple, but
exactly the kind of code that gets unmaintainable as rules accumulate.

**What I pushed for:** A pipeline of three pure functions (`classify` / `scheduling` /
`planActions`) glued by a 15-line orchestrator. The argument was that each layer
represents a real domain concept (the call's *meaning*, the *time math*, the *case state*)
and they should be testable independently. The AI agreed and produced the three approaches
table in `docs/design.md` §1.

**Why this matters:** I caught a smell that the AI defaulted to — "all logic in one
function" — before it became code. This is a small win that compounds: every later test
file is independently runnable because of it.

## §2 — Concrete disagreements during this session

### 2.1 Setup scope (resolved against the AI's first instinct)

The AI suggested running the full `ai-ready-project` skill — a multi-agent setup,
CLAUDE.md generated by template, hooks, the works. **I pushed back:** for a 2-hour
take-home where the reviewer's first impression is the repo, a 800-line CLAUDE.md and a
forest of `.claude/agents/*.md` *looks* like over-engineering, regardless of how useful
the tooling is for me. Final choice: minimal CLAUDE.md focused on TDD discipline + a
separate `dev-meta` branch for the full setup. (See README, "How to handle the meta-tooling".)

### 2.2 Date library

The AI's first thought on time arithmetic was native `Date` ("it's lightweight, no
dep"). **I pushed for Luxon** because the sujet explicitly calls out DST and
`Europe/Paris`, and `Date` has no concept of named zones — only offsets. A reminder
"09:00 Paris time on a date" is provably broken with native `Date` on DST days.
Verified by writing the DST test cases (`2025-03-30` → `07:00Z`, `2025-10-26` → `08:00Z`)
**before** the implementation, then watching them pass.

### 2.3 Audit log "sanitize" implementation (caught a real bug)

The AI's first regex for stripping control chars was `/[\r\n\t -]+/g`. **I read it
carefully and caught the bug** — the `-` in `[\r\n\t -]` is a literal character (or a
range from space to dash), which would also strip legitimate dashes from audit messages
like `wait_payment_confirmation`. Corrected to `/[\r\n\t\x00-\x1f]+/g` — explicit ASCII
control-char range. Tested.

### 2.4 Determinism contract enforcement

The AI's first scheduling helper called `DateTime.now()` internally for "convenience".
**I rejected it** — every test would have been non-deterministic. The rule "every
function takes `now` as an argument" is now hard-coded in `CLAUDE.md` so future sessions
can't regress it. See `src/scheduling.ts` — no clock reads anywhere.

### 2.5 Test that lied to itself

While writing the DST test, my draft used `NOW_SUMMER = "2025-06-15..."` with a payment
date of `"2025-04-15..."` — the payment was already in the past relative to "now". The
test expected a runAt **before now**, which contradicts the engine's "never schedule in
the past" rule. **The test was wrong, not the implementation.** Fixed the test to use a
`now` before the payment date. (Documented in commit history.)

## §3 — What I owned vs what I delegated

| Decision | Who decided |

| Dep choice: Luxon over native Date | **Me**, the AI initially preferred no-dep |
| Cascade order: telephony beats AI | **Verified by me** against sujet §1; the AI followed |

| Setup minimal vs ai-ready-project | **Me**, pushing back on AI default |

## §4 — What I manually verified

A reviewer is right to ask: "did you just trust the AI?" Here is what I actually checked
before committing.

1. **DST behavior on real boundary dates.** I cross-referenced the spring-forward
   (2025-03-30) and fall-back (2025-10-26) dates against the IANA tzdb. The tests
   assert the *exact* UTC instant (`07:00Z` vs `08:00Z`) so any silent fallback to
   wrong-offset would surface immediately.
2. **The classification cascade order matches the sujet.** Telephony safety overrides
   come first, then insights, then `callbackAt`, then early-termination, then unknown.
   I re-read sujet §1 ("Telephony Safety Overrides") to confirm this is the intended
   semantics — yes.
3. **No clock reads in `src/`.** I grepped: `grep -rE "(new Date\(|Date\.now)" src/` —
   returns nothing. The only `Date.parse(...)` calls are on input strings, never on
   wall-clock time.
4. **`noUncheckedIndexedAccess: true` in tsconfig.** Forces handling of `undefined` from
   array indexing — would have hidden multiple potential bugs in the cascade.
5. **Every test fails before its implementation.** TDD discipline meant I saw red for
   each test file before green. Listed in commit history (`feat(classify)`, `feat(scheduling)`,
   `feat(planActions)`, `feat: orchestrator`).
6. **The sanitize regex.** Regexes are exactly the kind of thing where an AI's "looks
   right" is wrong. Tested manually with a string containing newlines, tabs, and
   regular dashes; only the controls were stripped.
7. **The truncate behavior.** A 5000-char summary truncates to ≤ 2000; a 19-char summary
   passes through unchanged. The `…` suffix is appended; bytes match.

## §5 — Where AI saved me time (honest list)

- **Boilerplate** for test factories, `_factories.ts`, `package.json`, `tsconfig.json`,
  `vitest.config.ts` — these are 100% AI-generated and I just glanced at them.
- **Drafting the audit and warning messages.** Wording is AI; intent is mine.
- **Initial cascade implementation** in `classify.ts` after I gave the order.
- **Documentation writing speed** for `docs/business-rules.md` and `docs/edge-cases.md`.

## §6 — Where I would push back on AI again

- **Whenever an AI proposes a config-driven abstraction for a 2h problem.** Rule engines,
  DSLs, and plugin systems are seductive. They are almost always wrong for take-homes.
- **Whenever AI tests pass on first run without me seeing red.** Strong smell that the
  test is testing the implementation (tautology) rather than the requirement.
- **Whenever AI strips an explicit `now` argument for "cleaner" signatures.** Same as 2.4
  above — determinism is more important than ergonomics here.

## §7 — Honest limitations of this approach

- I'm one person reviewing AI output in real-time. A second reviewer would catch what I
  missed.
- The cascade has a lot of branches; a few corner cases (like B-B5, "very large
  durationSec") are *acknowledged* but *not tested* — documented in `docs/edge-cases.md`.
- The truncation cap is `2000` chars — picked by intuition, not measured against real
  production transcript sizes.
- I haven't run the engine against real Voxfit data. The "correctness" claim is about
  *internal consistency with the sujet*, not about real-world fitness.
